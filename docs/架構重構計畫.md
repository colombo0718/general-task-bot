## 一、現況問題

目前 GTB 專案資料夾混雜了「框架核心」與「各專案應用」：

```
general-task-bot/          ← 框架 + 所有專案混在一起
├── main.py                ← 核心框架 ✅
├── prompts.ini            ← 系統層提示詞 ✅
├── prompts_cs.ini         ← POS 客服專案設定 ❌ 不屬於這裡
├── prompts_pos.ini        ← 另一個專案設定 ❌
├── prompts_store.ini      ← 另一個專案設定 ❌
├── mission_cs.json        ← POS 客服任務定義 ❌
├── mission_pos.json       ← 另一個專案任務 ❌
├── mission_store.json     ← 另一個專案任務 ❌
├── customerlist.txt       ← 特定專案資料 ❌
└── todo_list.db           ← 特定專案資料庫 ❌
```

**目標：** GTB 只保留核心，各專案自帶設定與資料。


## 四、影子模式資料庫設計（shadow_db/shadow.db）

### 設計原則

- **一張表搞定**：一筆記錄對應一則客戶訊息的完整生命週期
- **一個客戶一張表**：表名用客戶識別碼命名，結構完全相同
- 用 SQLite，輕量不需額外伺服器

### 資料表結構（每個客戶建一張，結構相同）

```sql
-- 表名範例：messages_cwsoft、messages_客戶A
CREATE TABLE messages_{客戶識別碼} (
    id            INTEGER PRIMARY KEY AUTOINCREMENT,
    received_at   DATETIME DEFAULT CURRENT_TIMESTAMP,
    user_message  TEXT,     -- 客戶原始訊息
    intent        TEXT,     -- 意圖分類結果
    kb_sections   TEXT,     -- 引用的知識庫節次（如 "3-3,3-1"）
    ai_draft      TEXT,     -- AI 擬定的回覆草稿
    debug_hints   TEXT,     -- 工程師排查提示
    db_sp_refs    TEXT,     -- 涉及的資料表與 SP（JSON）
    staff_reply   TEXT,     -- 工程師實際回覆內容
    used_draft    BOOLEAN   -- 是否採用 AI 草稿
);
```

### 建表腳本（init_shadow_db.py）

```python
import sqlite3, os

DB_PATH = "shadow_db/shadow.db"
os.makedirs("shadow_db", exist_ok=True)

CREATE_TABLE = """
CREATE TABLE IF NOT EXISTS messages_{name} (
    id            INTEGER PRIMARY KEY AUTOINCREMENT,
    received_at   DATETIME DEFAULT CURRENT_TIMESTAMP,
    user_message  TEXT,
    intent        TEXT,
    kb_sections   TEXT,
    ai_draft      TEXT,
    debug_hints   TEXT,
    db_sp_refs    TEXT,
    staff_reply   TEXT,
    used_draft    BOOLEAN
)
"""

def init_customer_table(customer_name: str):
    conn = sqlite3.connect(DB_PATH)
    conn.execute(CREATE_TABLE.format(name=customer_name))
    conn.commit()
    conn.close()
```

## 五、LINE OA 訊息流程（影子模式）

```
客戶傳訊息（LINE OA）
        ↓
LINE Webhook → Flask（cwsoft-pos-knowledge-core/run.py）
        ↓
① 意圖分類（呼叫 GTB，使用 prompts_cs.ini）
② 選節次（GTB 第二輪，對照 kb-customer 目錄）
③ 讀取對應節次 md 檔
④ 產生回覆草稿（第一層知識庫）
⑤ 產生排查提示（第二層知識庫，若意圖為問題回報）
        ↓
寫入 shadow_db/shadow.db
        ↓
影子模式 UI 顯示草稿給客服
        ↓
客服確認 → 呼叫 LINE Messaging API 發送
```

**重點：** 客服不直接用 LINE OA Manager 回覆，而是透過影子模式 UI，確保每筆回覆都被記錄。

## 六、重構執行順序

### 第一階段：整理 GTB（不影響現有功能）
1. 建立 `config.example/` 資料夾，放入範例設定
2. GTB `main.py` 加入 `--project` 參數支援（向下相容，無參數時讀現有設定）
3. 逐步把 `prompts_cs.ini`、`mission_cs.json` 搬到 `cwsoft-pos-knowledge-core/config/`

### 第二階段：建立 POS 專案結構
1. 在 `cwsoft-pos-knowledge-core/` 建立 `config/` 與 `shadow_db/`
2. 執行 `shadow.db` 的建表 SQL
3. 寫 `run.py` 啟動腳本

### 第三階段：LINE OA Webhook 串接
1. `run.py` 加入 Flask webhook endpoint
2. 串接 GTB 產生草稿流程
3. 草稿寫入 `shadow_db`

### 第四階段：影子模式 UI
1. 技術選型（技術選型未定，候選：簡單 Flask 後台 / Streamlit）
2. 顯示 tickets + drafts，供客服審核後發送
