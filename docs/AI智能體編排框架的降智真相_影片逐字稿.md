如果你現在去問任何一個做前開發的資工程師，嘿，我們要怎麼去構建一個極其複雜的、絕不容許出錯的多智能體應用呢？
我敢打賭他們大概率會立刻在白板上給你畫出一堆複雜的流程圖，然後給你一個標準大，你需要一套極其大的交手架，這也就是大家說的外部 ent 的編排框架，比如說大明鼎底的 LANGHCWAI 或者是 Google 的 ATK 微軟的 smaticel。
在目前的行業普遍人之裡，構建 AI 應用就像是在蓋移動摩天大樓，所有人覺得大模型雖然聰明，但也是個難以馴服的充滿幻覺的野獸，要是沒有這些極其嚴密的基於狀態圖的一步步流程來控制框架，它絕對會像之無頭蒼蠅一樣徹底是控。
這幾乎已經成了當前 AI 開發界的絕對政治正確。
但是今天我要告訴你一個可能會徹底顛覆你常識的，甚至可能會讓你今晚就想把代碼庫刪了重寫的驚人事實：這個被整個科技界風為的開發鐵率，其實它可能正在讓你的 AI 變得越來越笨。
是的，你沒聽錯，這是一個極具破壞性的動察，今天我們需要來拆解一顆頭像當前 AI 工程飯食的重磅炸彈。
為什麼突然聊這個呢？
因為這關乎到我們每個人未來將如何構建，甚至如何使用 AI。
在過去的204年到2025年，各種 Aent 的框架如以後春損般爆發，但在繁榮的表相之下是無數開發者的痛苦哀嚎。
不知道你是否看過論文林引用的那份針對500多名開發者的大規模實政研究，數據相當慘烈，高達80%的人類工程師都在這些所謂的降低複雜性的編排框架裡痛苦掙扎。
我經常聽到開發者抱怨說，光為了給 AI 修改一個特別小的決策條件，他們竟然需要穿越整七層抽向代碼，跨越無數個節點，系統稍微一跑片就是一連串的連鎖錯誤。
我們設計這些框架的初衷明明是為了降低系統的複雜性，結果呢，反而在系統內部應生建起了一座根本沒法維護的代碼工。
就在這個行業陷入過度工程化的泥潭的時候，爾本大學的 Simon Dennis 教授團隊聯合幾位頂尖研究員在 archive 上丟出了一篇極其驚人的論文。
這篇論文的標題非常直接，甚至帶著點學術界罕見的火藥位，那就是**「上下文提示淘汰了程序性任務的智能體編排」。
各位請注意其中的用詞，當嚴謹的學者在論文標題裡直接使用淘汰這種極具攻擊性的詞彙式，這絕對不是什麼查於放後的學術切磋，這簡直就是對那些科技巨頭極力推崇的開發路徑的直接宣戰。
接下來就讓我帶著你一步一步去看清這幫研究員是如何用極器嚴謹的科學實驗拔下外部的編排框架的底庫的。
在進入精彩的實戰數據之前，我們先把概念對齊，這份研究到底是用什麼樣的科學方法去擊碎這座迷宮的呢？
研究團隊設計了一場極度嚴苛的雙忙對照實驗，他們使用了當前最頂尖的擁有龐大上小紋窗口的大模型之一 cloud 作為底層推離引擎。
請注意實驗控制的極其嚴謹，相同的模型，相同的測試場景，相同的評判標準，唯一的變量就是管理模型的方式。
他們對比的核心就是外部編排模式和一種一直被大家極度低估的上下文提示模式（income tax 模式）**。
為了讓你能直觀的感受這兩種模式的差異，我們來打深動的比方。
想像一下你現在要給一個擁有過目不忘能力並且極其聰明的超級快遞原指路，你讓它去完成一個極其複雜的連環成配送任務。
所謂的框架編排模式就好比系統是一個有著嚴重觀管理癖號的控制狂老闆，老闆手裡明拿著完整的城市全景地圖，但他死活呀不給快遞員看全貌。
為什麼呢？
因為老闆覺得快遞員腦容量不夠，怕他看全貌看暈了。
於是快遞元沒走到一個路口，老闆就發一條極短的新指令，告訴他現在你在 A 節點判斷一下天氣，然後告訴我左轉還是右轉。
等快遞員真的左轉過去了，老闆的系統再當一下，去後台數據庫查一下，計算出下一個節點，再發一條指令過去。
在每一個分查路口，老闆都要強行介入做一次額外的指令下法，這就是基於節點的外部編排。
而論文提出的 income tax 模式截然相反，在這個模式中系統是個極其懂得放權的大掌櫃。
他怎麼做的呢？
他直接把一張包含所有常規路線突發擁堵預案甚至是預備方案的超級地圖，用結構化的文本寫的清楚楚，一次性全部塞進快遞的腦子裡。
然後老闆對快遞說，兄弟，整個流程和所有可能發生的意外都在你腦子裡了，目標是把貨送到你自己看著辦，然後徹底放手，讓模型依靠自己及其大的全局判斷力和注意力機制去搞定整個流程，沒有額外部代碼來插手它的下一步。
聽到這裡，如果你是個傳統程序員，你本能的反應肯定是這怎麼可能，把幾十部的流程全塞進去，絕對會產生幻覺，絕對會跳步，絕對會忘記問用戶關鍵信息的。
可是接下來我要念出的實驗數據將徹底粉碎這個所謂的嘗試。
為了測試這兩番到底誰強，研究團隊輕挑選了三個急劇挑戰性、充滿了邏輯陷阱的真實商業場景，每個場景生成200次，一共進行了1200次極其嚴密的模擬對話。
這三場景分別是14個節點的 Zoom 技術支持，還有難度巔峰、整55個節點、跨有五個業務階段的汽車保險理賠處理。
他們不僅讓 Cloud 來生成對話，還使用一種叫 LM as a judge 的方法，讓模型作為無情的裁判，在任務成功率、信息準確性、邏輯一致性、優雅處理異常的能力，以及對話自然度這五個科的維度上進行一到五分的盲品。
1200次對決之後數據長什麼樣呢？
我直接告訴你結論，反差極其驚人，在總共15個對比項中，income tax 模式取得了令人難以性的15戰全勝，絕對的統治級碾壓，所有的差距在統計學上都具備顯著意義。
我們不要觀看別的數據，洛音裡附的真實對話日製才是最讓人拍的，讓我們來解剖三個框架翻車的車禍現場。
第一個現場 Zoom 技術支持的鬼打槍。
在這個場景下，用戶的麥克風沒有聲影，如果你用外部框架去管理，失敗率是多少呢？
是高達9%。
而讓 AI 自己拿全流程圖去思考呢，失敗率只有極低的0.5%。
為什麼會這樣呢？
論文力展示了一段極其經典的 LANGG 翻車對話，用戶說我在用 Max 上的 Crow 瀏覽器開 Zoom 別人聽到我說話。
確實在架的微觀管理下，斯卡在了當前的系統設置排查節點，他生意的讓用戶去查 m 的系統 m 風設置，用戶查的說沒問題。
這時候如果是個人類客服，肯定會馬上反應過來說，哦，那可能是 Chrome 瀏覽器的權限沒開。
但是 LFY 的 AI 此時是個爭眼下，因為它只被為了當前節點的 prompt，它根本無法調動全局視角。
於是他幹了什麼呢？
他開始讓用戶點擊測試麥克風大點聲說話，在原有的排障節點裡陷入了死循環，他喪失了跳出當前節點進行綜合診斷的靈性。
反官 incomex 模式，因為 AI 一開始就把所有的排賬可能都裝在了腦子裡，他在第二回合就極其敏銳的指出，哦 Mac 配網頁版這很可能是 MacOS 隱藏的瀏覽器 M 麥克風權限問題，你看一機致命，六個回合解決戰鬥。
第二個現場旅遊預定的機械降聲。
我們再看一個旅遊預定的例子，場景是四個好朋友去西班牙玩九天，預算是2600歐元，想要極限運動和精品酒店。
在框架編排模式下，L 用戶聊得好好的，但因為信息被節點強行切碎，當系統從人數確認節點切換到預算評估節點，這中間經歷了一次所謂的決策路由史，AI 的上下文出現了一個致命的邏輯斷層。
AI 突然死的認為這2600歐元是個人九天的總預算，於是極其荒謬的一幕出現了，AI 像個沒有感情的機器陽對用戶說，您好四個四個人2600歐要包含機票時速精品酒店是絕對不可能的，這邊建議您換個需求。
用戶及了趕緊解釋，等我說的是每人2600歐不是總合。
這時候框架雖然強行把對話拉了回來，但這種因為節點切換而導致的關鍵信息丟失，不僅浪費了大量的對話回合，還讓用戶體驗極其。
而在 income tax 的模式下從頭到為掌控，他不僅沒有算錯賬，還像個真正的高級定制導遊一樣，天無縫的把用戶想要的峽谷跳傘和海景精品酒店交織在了一起，自然流暢的讓人起疙瘩。
第三個現場55節點保險理賠的末日探 ately 塔。
如果說前兩個常用鏡只是讓人生氣，那在擁有55個節點、六個決策樞紐的汽車保險理賠場景裡，框架編排是一場災難。
在這場景裡，用戶半夜在鄉村公路撞到了一頭路，這種理賠涉及報案定損免賠額算租車補償等極其複雜的欠套邏輯。
在55個節點的劇型圖裡，L 出現了恐怖的錯誤級連效應，因為決策樞紐太多了，只要在某一個路口框架調用大模型做了一次錯誤的路線選擇，這個錯誤就會像雪崩一樣下游傳遞，吸一同會開始跳躍階段或者瘋狂的像用戶所要已經提供過的證件。
面對這麼複雜的場景的表現如何呢？
那是令人恐懼的完美啊，當把這含有55個節點大約4000個 token 的巨型結構化流程圖一次性餵給大模型時，大星不僅沒有暈，反而展現出了極其大的整體規劃能力，它由刃有餘的完成了所有的階段，邏輯連貫性得分高達4.83，而框架模式只有可憐的4.39。
讀到這裡，你可能會覺得渾聲發，為什麼呢？
因為我們全行業住了海量的資金和幾萬個日夜打造的 A 陣的框架，竟然被一個簡單的把所有提示詞塞進去的粗暴方法按在地上摩擦。
框架到底是怎麼把事情搞砸的呢？
研究人員在論文的 discussion 部分及其尖銳的指出了框架編排的三大元罪，簡單來說就是我們用管理傳統死板代碼的傲方式去強行管理了一個基於概率的擁有湧現能力的認知系統。
第一大元罪是推理碎片化，這是最致命的一擊。
大模型的核心智慧恰恰來源它通過海量的上下文在詞與詞之間極其微妙的概率分佈和注意力聯繫，當你用框架把它一刀一刀切斷每次只給他當前節點的局部信息時，他就像被蒙住了眼睛。
他只能在極度狹窄的窗口裡做出在局部看似合裡但在全局卻極其荒謬的決定，他失去了對話整體胡光的感知。
第二大元罪是引入了全新的失敗模式，大家用框架是為了求穩，結果框架自己成了最大的 bug 製造機。
論文指出外部框架額外製造了三種本來不該存在的錯誤：首先是路由失敗，當人類的話語充滿灰度時框的死板的 if else 邊條件根本無法匹配導致系統走入死胡同；其次是角色奇異，用戶一句話可能同時出發了兩個節點的條件，框架不知道該怎麼選直接卡死；最後是模板衝突，框架強行注入的當前節點規則跟用戶之前的聊天記錄發生了衝突，本來是為了求穩而引入的中間商反而成了最大的系統崩潰源頭。
第三大元罪也是最讓人惜的一點，他徹底殺了模型的自然才華，這在論文的 N 評分中體現的淋漓禁制。
這就好比你花大家錢請了一位決頂聰明的控場能力極強的脫口秀天才，然後給他套上射了一個冰冷的提刺器，強制他必須按格式念稿，這簡直是暴天舞。
框架那種基於節點的模板注入，硬生的把原本可以華溝通的前言大模型降為成了一個充滿機械感的只會疑問一達的智障科夫機器人。
當然作為嚴謹的科學家 DIS 教授團隊也考慮到了這一點，由於生成對話和當裁判的都是 Cloudson 會不會存在一種自戀偏差呢？
也就說 CL 天生就喜歡這種沒有框架約束自由自在生成的文字呢。
為堵住所有人的嘴，研究團隊做了一件其硬和的是，他們花了大量的 API 費用找來了 OPI 的 GPT4.1用一模一樣的標準把這1200場對話重新打分忙平了一遍。
結果呢，核心結論兼如盤時，在任務成功率信息準確度邏輯一致性上，CP4.1依然判定 income tax 模式呈現壓倒性優勢。
雖然在自由度上 GPT4.1覺得兩者差不多，但框架模式在執行任務上的低效和混亂已經被兩家最頂尖的大模型交叉正式再無翻盤的可能。
聽到這裡，作為一線的工程師或者企業的 CTO 你腦子裡肯定緊靈大作，你一定在想一個其現實的問題：錢。
你可能會反駁我等，如果不搞節點切分，難道用戶每發一句話我都要把幾千字的55節點流程圖完整的塞進大模型的腦子裡嗎？
在按 token 的今天，這難道不是在瘋狂燒前嗎？
框架模式雖然笨但他每次傳給大模型的上下文很短啊，這樣難道那不是能省錢嗎？
各位，這恰恰是這項研究中最反直覺也最精妙的經濟學發現，這就是我們在 AI 開發中常犯的只見數目不見森林的錯誤。
表面上看 income text 的模式單次塞的字數確實多了點，但如果我們算一筆全練系統總賬，結論會完全顛覆你的直覺。
大家總以為切分節點能省 AP 調用次數，但是正好相反，論文的表三接了一個可怕的真相，因為編排框架在每個決策樞紐都需要額外調用一次大模型來做路由判斷，這就導致了極其龐大的不用開銷。
數據表明完成一次複雜的汽車保險理賠，框架模式平均需要調用 API 高達17.3次，而自己掌控全局的 income 模式呢只需要區10次。
這意味著什麼呢？
這意味著如果你有框架，你的用戶要在這個業務流程中多承受七次極其漫長的網絡等待和推理延遲折磨，在 C 端產品裡多一秒的延遲，用戶的流失率就會直線上升。
那落到真金白銀上呢，論文也算得清清楚楚，由於要把整個流程圖塞進去，income tax 模式處理一次55節點的保險理賠消耗了約6.8磅個 token 成本是0.22美元，而框架模式雖然省 token 但也花了0.17美元，兩相差多少呢？
趨區5美分。
所以你願意為了每次對話省下這5美分去忍受多達70%的 API 網絡延遲去承擔高達17%的業務失敗率去犧牲價值幾百上千美元的客戶體驗甚至導致客戶因為客服太閃而推保嗎？
在這筆賬面前任何一個有基本商業邏輯的人都會發現，為了省而過度切分節點絕對是典型的減了芝麻丟了西掛。
陸文詞不會的說，在這些任務中性能的瓶頸根本不是不夠好的編排，編排本身就是那個最大的平頸。
當然永遠要面對真實生產環境的即刻，我們必須保持冷的批判性思考，我們不能被實驗數據的狂歡衝寬的頭腦失去現實的引力。
所以這篇論文是在號召我們燒毀世界上的所有的 Aent 的框架 l 從 GHub 上徹底刪除嗎？
絕對不是。
這篇論文極其克制的在第5.3節化定了他的邊界，它的15勝零負是大前提的，了了解這些前提才是區分普通馬農和頂級 AI 架構式的關鍵，在以下這幾種情況裡，外部編排框架依然有著不可替代的保命價值。
