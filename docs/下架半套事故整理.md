# 下架半套事故整理

## 背景

在阿全經理新專案切換到 `gtb.py` 後，系統內嵌的 todo worker 開始巡查該專案自己的 `database/todo_list.db`。

巡查過程中，worker 撈到了過去遺留的 `pending` 任務，其中包含高風險的客戶下架操作，例如：

- `https://cwsoft.leaflune.org/sqlgate/detach_customer?name=千訊`

系統 log 顯示：

```text
[TODO_WORKER] executing id=4
[TODO_WORKER] failed id=4 error=404 Client Error: Not Found for url: https://cwsoft.leaflune.org/sqlgate/detach_customer?name=千訊&user_id=...
```

之後進一步檢查資料庫 `千訊` 的實際狀態，發現：

- `posconfig.停用 = 0`
- `停用時間 = NULL`
- `[各資料庫設定].dbo.資料庫設定檔` 仍存在
- 正式資料庫 `千訊` 仍為 `ONLINE`

這表示此次並未造成完整下架成功，但也暴露出「高風險任務可能在未知狀態下被重新執行」的問題。


## 這次為什麼會發生

### 1. 舊的 pending 任務被新的內嵌 worker 撈出來

`gtb.py` 內嵌的 todo worker 會定期查詢：

- `state = 'pending'`
- `run_at <= now`

因此，只要資料庫中還留有舊的 `pending` 任務，worker 一啟動就可能去執行。

這次阿全專案的 `todo_list.db` 裡確實存在多筆舊任務，其中包含：

- 過期的 `detach_customer`
- 過期的 `adjust_points`
- 一筆 `run_at` 是整段中文分析文字的異常資料

也就是說，**事故的直接觸發點不是使用者重新下指令，而是舊排程殘留被重新巡到。**


### 2. `pending` 狀態的語意過於寬鬆

目前系統中：

- `pending` 代表尚未完成
- 也代表執行失敗後仍待重試

這會造成一個問題：

- 高風險任務只要還是 `pending`
- 且 `run_at <= now`
- worker 就會不斷嘗試重跑

目前沒有額外的：

- `running`
- `failed`
- `last_error`
- 最大重試次數

所以錯誤任務可能被反覆嘗試。


### 3. `run_at` 缺乏嚴格格式驗證

在 `todo_list.db` 中發現一筆異常資料，其 `run_at` 不是合法時間，而是一整段模型分析文字。

這代表目前排程寫入流程缺乏足夠的驗證：

- 不合法的 `run_at` 仍可能被寫入資料庫
- worker 撈任務時是用文字比較，不是嚴格 datetime 驗證

結果就是：

- 任務是否會被撈到，可能不符合直覺
- 一旦被撈到，就可能執行危險操作


### 4. 高風險任務沒有額外保護

像 `detach_customer` 這類操作，本質上屬於高風險指令，因為它可能：

- 停用客戶
- 刪除設定資料
- 卸載正式資料庫
- 卸載測試資料庫

但在當前架構中，這類任務仍和一般任務共用相同的排程機制，缺乏額外保護，例如：

- 強制二次確認
- 風險提示
- 操作審計
- 執行前再次檢查


### 5. 預存程序本身也存在「半套狀態」風險

`DetachCustomer` 這支預存程序本身不是 transaction-safe。

它會依序執行：

1. 更新 `posconfig` 停用狀態
2. 刪除資料庫設定檔
3. 刪除聯盟客戶資料
4. 停用行動裝置
5. `sp_detach_db` 卸載正式資料庫
6. `sp_detach_db` 卸載測試資料庫

但整個過程沒有：

- `BEGIN TRAN`
- `COMMIT`
- `ROLLBACK`
- `TRY/CATCH`

這代表只要中途任何一步失敗，就可能留下：

- 部分資料已刪
- 部分狀態已改
- 但資料庫未完全 detach

也就是所謂的「只下架一半」。


## 為什麼這次看起來沒有完全傷到 `千訊`

從目前查到的結果來看：

- 正式資料庫 `千訊` 仍然 `ONLINE`
- `posconfig` 沒被標停用
- 設定檔還在

這與 worker log 中的 `404 Not Found` 是一致的。

較合理的推論是：

- 這次請求在 HTTP 層就失敗了
- 真正危險的 SQL 沒有完整執行到底

但這只是「這次剛好沒打中最壞情況」，不代表架構本身安全。


## 事故的根本原因總結

這次會出現「下架半套」風險，根本原因不是單一 bug，而是多個問題疊加：

1. 舊 `pending` 任務殘留
2. worker 啟動後自動重跑過期任務
3. `run_at` 缺乏嚴格驗證
4. 高風險任務沒有額外保護
5. 預存程序本身不具交易保護


## 已採取的立即處置

目前已做的止血動作：

1. 將阿全專案 `todo_list.db` 中的舊 `pending` 任務改為 `done`
2. 將 AI 客服專案 `todo_list.db` 中的舊 `pending` 任務改為 `done`
3. 修正 `gtb.py` 中 webhook 的 `profile` 變數錯誤
4. 將 `gtb.py` 中阿全 POS 特化邏輯移除


## 後續建議

### GTB 層

1. 為 `run_at` 加入嚴格格式驗證
2. 將排程狀態擴充為：
   - `pending`
   - `running`
   - `done`
   - `failed`
3. 失敗任務不要無限重試
4. 建立高危險任務預警系統
5. 建立完整操作審計紀錄

### 任務層

1. 高風險任務在 mission 中加風險標記
2. 高風險任務強制雙重確認
3. 高風險任務最好不要直接走自動排程

### SQL / 後端 API 層

1. `DetachCustomer` 改成 transaction-safe
2. 補 `TRY/CATCH + ROLLBACK`
3. 正式庫與測試庫 detach 前都做存在檢查
4. 執行前先寫操作 log
5. 成功與失敗都回傳明確狀態


## 結論

這次事件雖然沒有造成 `千訊` 完整下架，但已清楚暴露出：

**GTB 的舊排程殘留 + 高風險任務缺乏保護 + 後端預存程序不具交易保護，三者組合起來，足以造成「只下架一半」甚至更嚴重的事故。**

因此，後續必須同時從：

- GTB 排程機制
- 高危險任務保護
- SQL 預存程序安全性

三個層面一起補強。
